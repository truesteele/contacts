## US-005: RSS Enrichment Script â€” COMPLETE
- Created `scripts/intelligence/enrich_podcasts.py`
- Pipeline: load un-enriched â†’ fetch RSS (concurrent, 20 workers) â†’ parse XML â†’ extract host email/name â†’ parse last 5 episodes â†’ classify activity â†’ optionally verify email with ZeroBounce â†’ save
- Supports `--limit`, `--workers`, `--skip-verify`, `--test` flags
- RSS parsing: `xml.etree.ElementTree` with iTunes namespace (`http://www.itunes.com/dtds/podcast-1.0.dtd`)
- Email extraction: `<itunes:owner><itunes:email>`, fallback to `<itunes:email>` on channel
- Noreply filtering: regex filters out `@anchor.fm`, `@mg*.acast.com`, `noreply`, `donotreply`, `abuse@`
- Activity classification: active (<30 days), slow (30-90 days), podfaded (>90 days)
- Episode parsing: title, description (HTML stripped), pubDate (RFC 2822), duration (HH:MM:SS or seconds), enclosure URL
- ZeroBounce: reuses verify_email pattern from find_emails.py (credit check, retry, rate limit handling)
- Tested on 5 iTunes-discovered podcasts: 2 real emails found, 3 generic emails filtered, 25 episodes saved, 1 active / 4 podfaded
- Re-enrichment safe: deletes old episodes before inserting new ones

### Learnings
- Podcast RSS feeds use `<itunes:owner><itunes:email>` for host contact info â€” most reliable source
- Many podcast hosting platforms (Anchor/Spotify, Acast) inject their own generic addresses â€” must filter
- `email.utils.parsedate_to_datetime` handles RFC 2822 dates from `<pubDate>` cleanly
- iTunes duration can be raw seconds OR HH:MM:SS format â€” need to handle both
- Categories in RSS use `<itunes:category text="...">` attribute (not text content) with nested subcategories
