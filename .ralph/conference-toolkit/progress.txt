### Iteration 3 — US-003: Config-driven triage script (2026-04-10)

**What it does:**

- Same concurrent GPT-5 mini scoring: ThreadPoolExecutor, structured output, 3x retry
- Saves results with field-prefix-aware keys (ted_id, ted_name)
- Supports `--test`, `-n`, `--workers` flags

**Verified:**
- `--test` with 3 attendees: 3/3 scored successfully, 0 errors
- Output format matches existing `ted_triage_results.json` exactly (ted_id, ted_name, relevance_score, partnership_type, etc.)

**Key decisions:**
- Dynamic Pydantic enum built at runtime from config partnership_types — GPT output is constrained to valid types
- Field prefix from config drives all attendee field lookups — no hardcoded `ted_*` assumptions
- User connection fields derived from config (primary/support connection_field) with fallback to `{name.lower()}_connection`
- Default `-n 3` in test mode (original was 10) — faster iteration

### Iteration 4 — US-004: Config-driven populate script (2026-04-10)
**Status:** COMPLETE
**Commit:** `feat: [US-004] Conference toolkit - Config-driven populate script`
**Files created:**
- `scripts/conference/populate.py` — Generic config-driven Supabase population

**What it does:**
- Accepts `--config` CLI arg pointing to any conference config YAML
- Loads warm_leads, triage_results, shortlist from config data_paths
- Joins all three by the config-derived ID field (e.g., `ted_id`)
- Auto-detects and copies all lead fields, boolean-ifies role fields and connection fields
- Adds triage scoring fields (relevance_score, partnership_type, etc.) and tier from shortlist
- Initializes user-specific columns from config (pinned, reached_out, notes, context) with defaults
- Migrates existing state from legacy `{prefix}_lookbook_state` table
- Upserts in configurable batches (default 500) with on_conflict on the ID field
- Reports summary stats: tier counts, scored counts, partnership type distribution
- `--dry-run` shows prepared rows and sample without touching DB
- `--batch-size N` for custom batch sizes

**Verified:**
- `--dry-run` with TED 2026 config: 1,884 rows prepared, 84 with tiers, correct field mapping
- Sample row has all expected keys (ted_id through justin_context)

**Key decisions:**
- Auto-copy all lead fields rather than explicit mapping — more generic, handles any field prefix
- Legacy state migration tries `{prefix}_lookbook_state` table, silently skips if not found
- User columns driven entirely by config `users.*.columns` dict — no hardcoded sally_/justin_ assumptions
